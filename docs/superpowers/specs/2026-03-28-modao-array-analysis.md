## 1. 项目概况

### 1.1 基本信息
- **项目名称**: 魔道塔防 (Modao Tower Defense)
- **仓库**: https://github.com/cooolinx/modao-array
- **技术栈**: Pixi.js v8+ (WebGL), ES Module, Web Audio API
- **游戏类型**: H5 塔防
- **当前状态**: 核心玩法可玩 (v0.1)

### 1.2 核心玩法
玩家扮演万劫魔宫主人（万劫老祖），抵御六大宗门围攻：
- 建造防御塔（3 种）抵御敌人
- 使用「万鬼大阵」技能召唤鬼军助阵
- 关卡制推进（7 章 25 小关）
- localStorage 存储通关进度

## 3. 问题识别

### 3.1 高优先级问题（影响核心体验）

#### P1: 塔系统策略深度不足
**问题描述**:
- 仅 3 种塔，缺乏多样性
- 塔无法升级，缺少成长感
- 塔无法出售，缺少纠错机制
- 无特殊效果塔（穿透、连锁、DOT）

**影响**: 玩家策略选择有限，重复游玩价值低

**设计规范**（补充）:
- **升级系统**: 3 级上限，线性升级
  - 升级成本：50/75/100（基础塔），100/150/200（炮台），75/100/125（寒冰阵）
  - 升级效果：伤害 +50%/级，攻速 +20%/级，射程 +15%/级
- **出售系统**: 返还 60% 总成本（包括升级投入）
- **视觉反馈**: 每级塔外观变化（颜色/大小/装饰）

#### P2: 敌人系统单一
**问题描述**:
- 仅 3 种基础敌人
- 无 Boss 单位，缺乏高潮
- 无飞行单位（无视路径）
- 无特殊技能敌人（隐身、自爆、治疗）

**影响**: 战斗体验单调，缺乏挑战性

**Boss 战设计框架**（补充）:
- **Boss 类型**: 每章最终波出现 1 个 Boss
  - 第 1 章 Boss: 青云剑阁·剑尊（高血量、剑雨技能）
  - 第 2 章 Boss: 天涯书院·儒圣（召唤书生、治疗技能）
  - ...（6 大宗门对应 6 个 Boss）
- **Boss 机制**:
  - 血量：普通敌人的 10-20 倍
  - 技能：每 10 秒释放一次特殊技能
  - 阶段转换：血量 50% 时进入狂暴状态（速度 +50%）
- **击败奖励**: 大量灵石（200-500）、解锁下一章

#### P3: 游戏控制功能缺失（已升级为核心功能）
**问题描述**:
- 无暂停功能
- **无游戏速度控制（1x/2x/3x）** ← 塔防游戏标准配置
- 无塔射程预览

**影响**: 后期等待时间长，操作体验差

**备注**: 根据审查意见，游戏速度控制是塔防游戏（如《王国保卫战》《植物大战僵尸》）的标准功能，应从 P3 提升至 P1。

#### P4: 视觉反馈弱
**问题描述**:
- 敌人死亡无特效
- 无伤害数字飘字
- 无屏幕震动
- 塔攻击动画单一

**影响**: 打击感不足，爽快感低

### 3.2 中优先级问题（功能完善）

#### P5: UI/UX 信息不足
- 塔按钮无 DPS/射程/攻速显示
- 无暂停按钮
- 无音量控制

#### P6: 音效系统不完善
- 无背景音乐
- 程序化音效单调
- 无音量滑块

#### P7: 缺少游戏模式
- 无星级评价
- 无无尽模式
- 无成就系统

#### P8: 缺少工具
- 关卡编辑器未完成（map-editor.html）
- 路径/塔位手动计算坐标繁琐

### 3.3 低优先级问题（锦上添花）

#### P9: 剧情/叙事缺失
- 无章节过场
- 宗门背景故事未展现

#### P10: 美术资源粗糙
- 使用 SVG/程序图形
- 各章背景图未制作

#### P11: 技术债务
- main.js 过于庞大（~500 行）
- 无 TypeScript 类型定义
- 无单元测试
- 无构建工具

## 4. 优化方案设计

### 4.1 方案 A: 核心体验优先（推荐）

**重点**: 先完善核心玩法，再扩展内容

**经济系统设计**（补充）:
- **金钱获取**:
  - 击杀普通敌人：10-20 灵石（随波次递增）
  - 击杀 Boss:200-500 灵石
  - 初始资金：150 灵石
- **塔造价**:
  - 骷髅弓手：50 灵石
  - 魔炮台：100 灵石
  - 寒冰阵：75 灵石
- **经济曲线**:
  - 前期（波次 1-3）: 收支平衡，鼓励合理布局
  - 中期（波次 4-6）: 略有盈余，可升级塔
  - 后期（波次 7-8）: 经济宽松，追求最优配置
- **利息系统**（Phase 3）: 每波结束时，每 10 灵石额外奖励 1 灵石（上限 50）

**阶段划分**:
```
Phase 1 (1-2 周): 核心体验完善
  └─ 塔升级/出售系统
  └─ 游戏速度控制
  └─ 暂停功能
  └─ 伤害飘字 + 死亡特效

Phase 2 (2-3 周): 内容扩展
  └─ Boss 单位设计
  └─ 飞行敌人
  └─ 2-3 种新塔

Phase 3 (1-2 周): 系统完善
  └─ 星级评价
  └─ 无尽模式
  └─ 音效完善

Phase 4 (1 周): 工具优化
  └─ 关卡编辑器
  └─ 对象池优化
```

**优点**:
- 快速提升核心体验
- 风险低，每阶段可独立交付
- 符合 TDD 流程

**缺点**:
- 内容扩展较慢
- 美术资源滞后

**复杂度**: 中等
**预计工期**: 5-8 周

**优点**:

**缺点**:
- 核心体验问题延后
- 新内容可能因体验问题被弃用

**复杂度**: 高
**预计工期**: 6-8 周

### 4.3 方案 C: 重构优先

**重点**: 先重构代码架构，再开发新功能

**阶段划分**:
```
Phase 1 (2 周): 架构重构
  └─ 迁移到 TypeScript
  └─ 引入 Vite 构建
  └─ 添加单元测试框架

Phase 2 (2 周): 核心功能
  └─ 塔升级/出售
  └─ 游戏速度控制

Phase 3 (2 周): 内容扩展
  └─ 新塔/新敌人
  └─ Boss 战
```

**优点**:
- 代码质量高
- 长期维护成本低
- 类型安全

**缺点**:
- 前期投入大
- 用户可见改进慢
- 可能过度工程化

**复杂度**: 高
**预计工期**: 6-8 周

## 5. 推荐方案

### 5.1 推荐：方案 A（核心体验优先）

**理由**:
1. **风险最低** — 每阶段独立可交付
2. **用户体验提升快** — 优先解决痛点
3. **符合敏捷原则** — 小步快跑，快速迭代
4. **技术债务可控** — 在开发中逐步优化

### 5.2 关键决策点

| 决策 | 选择 | 理由 |
|------|------|------|
| 是否迁移 TypeScript | 暂缓 | 当前 JS 可维护，优先功能 |
| 是否引入构建工具 | 暂缓 | CDN 加载满足需求 |
| 是否添加单元测试 | Phase 3 开始 | 核心稳定后开始 |
| 是否先做美术 | 否 | 程序图形可接受，功能优先 |

### 5.3 成功标准（量化版）

- ✅ **塔升级/出售系统**: 左键点击塔可升级（显示升级成本），右键出售（返还 60% 成本），升级后伤害/攻速/射程按倍率提升
- ✅ **游戏速度控制**: 支持 1x/2x/3x 无缝切换，切换时游戏逻辑速度同步调整（子弹、敌人、特效速度一致）
- ✅ **伤害飘字**: 敌人被击中时显示伤害数字，向上飘动 2 秒后消失
- ✅ **死亡特效**: 敌人死亡时触发粒子爆炸（8-12 个粒子），1 秒后消失
- ✅ **暂停功能**: 按 P 键暂停游戏，所有游戏逻辑停止（子弹、敌人、特效冻结）
- ✅ **Boss 战**: 第 1 章最终波出现 Boss，血量 2000+，有技能动画，击败后解锁第 2 章
